# 开发环境

> 版本：v0.5.10-econ | 更新：2026-03-14

## 打包前提

1. Node.js 已安装，`frontend/` 下 `npm install` 已执行
2. `frontend/resources/python/` 里有完整的 embedded Python 3.12 环境（含 `requirements.txt` 依赖）
3. 开发环境的后端（8000/18081 端口）已关闭，避免误连本机开发服务

## 一、准备 embedded Python

首次准备、切换 Python patch 版本、或 `requirements.txt` 有变更时执行：

```bash
scripts\setup_python.bat
```

该脚本会：
- 下载固定版本的 Windows embeddable package（当前固定为 Python 3.12.10 x64）
- 清理旧的 `frontend/resources/python/`，避免 3.11 / 3.12 文件混杂
- 启用 `python312._pth` 中的 `import site`
- 使用 `get-pip.py` 安装 pip
- 安装 `requirements.txt`
- 调用 `scripts/prune_embedded_python.py` 裁剪非运行时文件

准备完成后，至少确认以下文件存在：

- `frontend/resources/python/python.exe`
- `frontend/resources/python/python312._pth`

可用以下命令做最小验证：

```bash
frontend\resources\python\python.exe --version
frontend\resources\python\python.exe -c "import pandas, numpy; print('ok')"
```

## 二、打包顺序

每次需要发布安装包时，按以下顺序执行：

```bash
scripts\setup_python.bat
build_backend.bat
cd frontend
npm run electron:build
```

产物：
- `frontend/release/econ-agent Setup x.x.x.exe` 或同类 NSIS 安装包
- `frontend/release/win-unpacked/` — 解压版（可直接运行，用于调试）

## 三、关键配置说明

### package.json build 字段

`frontend/package.json` 已声明：

- `resources/python -> resources/python`
- `../dist/backend -> resources/backend`

因此本轮重点不是改 electron-builder 配置，而是确保 `frontend/resources/python/` 在打包前真实存在且内容完整。

### 打包态解释器注入链路

- `frontend/electron/main.cjs:getPythonPath()`：打包态返回 `process.resourcesPath/python/python.exe`
- `frontend/electron/main.cjs:startPython()`：启动 `resources/backend/backend.exe` 时注入 `PYTHON_EXECUTABLE`
- `src/tools/code_runner.py:_find_python()`：优先读取 `PYTHON_EXECUTABLE`，其次 fallback 到 `resources/python/python.exe`

这意味着：
- 后端主进程与 `run_python` 子进程都应优先使用安装包内置解释器
- `run_python` 不应依赖系统 PATH 上的 Python

## 五、验收标准

### A. 源目录验证

确认以下文件存在：
- `frontend/resources/python/python.exe`
- `frontend/resources/python/python312._pth`

### B. 打包产物验证

确认以下文件存在：
- `frontend/release/win-unpacked/resources/backend/backend.exe`
- `frontend/release/win-unpacked/resources/python/python.exe`

必要时抽查 `resources/python/` 下关键运行时文件是否被带入，例如：
- `python312.dll`
- `python312._pth`
- `Lib/site-packages/` 中核心依赖

### C. 运行时验证

优先运行 `win-unpacked` 版，并确认：
- Electron 能正常拉起 `resources/backend/backend.exe`
- `main.cjs` 注入的 `PYTHON_EXECUTABLE` 指向 `resources/python/python.exe`
- `run_python("print(1+1)")` 类最小请求可正常返回
- 即使系统 PATH 中没有 Python，也不影响打包态执行 Python 工具

## 六、踩坑记录

### 1. 只重打前端、不重打 backend.exe

**问题**：后端逻辑变更后若不先执行 `build_backend.bat`，安装包仍可能携带旧版 `backend.exe`。
**解决**：始终遵循 `setup_python -> build_backend -> electron:build` 的顺序。

### 2. resources/python 目录缺失

**问题**：`frontend/package.json` 虽已声明 `extraResources`，但若 `frontend/resources/python/` 本身不存在，安装包里不会凭空生成 `resources/python/python.exe`。
**解决**：发版前强制执行 `scripts/setup_python.bat`，并在源目录与 `win-unpacked` 两处都做存在性检查。

### 3. Embedded Python 版本漂移

**问题**：若脚本只写主次版本或重复执行不清理旧目录，容易出现 3.11 / 3.12 文件混杂。
**解决**：脚本固定到明确 patch 版本，并在每次准备前清理 `frontend/resources/python/`。

### 4. run_python 误回退到系统解释器

**问题**：若未向后端注入 `PYTHON_EXECUTABLE`，`code_runner.py` 可能 fallback 到其他解释器。
**解决**：保留 `main.cjs -> PYTHON_EXECUTABLE -> code_runner._find_python()` 的优先链路，并在打包态做一次最小 `run_python` 验证。
