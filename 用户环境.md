# 用户环境

本文档用于统一 `econ-agent` 的三类环境认知：**开发环境**、**打包环境**、**最终用户环境**。

重点不是把所有本地文件都提交到 git，而是明确：**哪些配置虽然不会进仓库，但仍然会直接影响首启安装、模型可用性、打包产物和线上排障结果，必须保持一致认知。**

---

## 1. 运行时路线结论

当前项目采用以下稳定路线：

- **embedded Python 3.12**：应用运行时本体
- **uv**：给 embedded Python 安装依赖的首选安装器
- **pip**：uv 失败时的 fallback

不采用“让 uv 自己管理产品内 Python runtime 本体”的方案，原因：

1. 当前打包链路已经围绕 `frontend/resources/python/python.exe` 建立。
2. `frontend/electron/main.cjs` 在打包态会显式设置 `PYTHON_EXECUTABLE=process.resourcesPath/python/python.exe`。
3. `src/api/dependency_installer.py` 已按“固定目标解释器 + uv/pip 装包”模式工作。
4. 若改成 uv-managed runtime，会引入新的 bootstrap、路径恢复、离线分发与排障复杂度，不符合当前“小改高成功率”目标。

---

## 2. 开发环境

开发环境的目标是：**便于调试、便于改代码、不要污染系统全局环境。**

### 2.1 Python

推荐使用项目根目录本地虚拟环境：

```bash
python -m venv .venv
.venv\Scripts\activate
uv pip install -r requirements.txt
```

约定：

- 开发态优先使用仓库根目录 `.venv`
- 不依赖系统全局 Python 作为日常开发主环境
- `frontend/electron/main.cjs` 在开发态会优先寻找 `.venv/Scripts/python.exe`

### 2.2 Node / 前端

```bash
cd frontend
npm install
npm run dev
npm run electron:dev
```

约定：

- 默认使用 **Node.js + npm**
- `frontend/node_modules/` 为本地开发产物，不提交 git

### 2.3 开发态数据目录

开发态主要使用仓库内数据目录：

- `./data/memories.db`
- `./data/checkpoints.db`
- `./data/settings.json`
- `./data/workspace/`

说明：

- 这套数据主要用于本地调试
- 不应把个人测试数据、真实 API key、临时工作区内容提交到 git

## 3. 打包环境

打包环境的目标是：**生成一个不依赖用户本机 Python 的安装版。**

### 3.1 embedded Python 准备

通过以下脚本准备运行时：

- `scripts/setup_python.bat`

它会完成：

1. 下载 Python 3.12.10 embedded package
2. 解压到 `frontend/resources/python/`
3. 打开 `site-packages`
4. 安装 `pip`
5. 安装 `uv`

最终关键文件：

- `frontend/resources/python/python.exe`
- `frontend/resources/python/Scripts/uv.exe`

### 3.2 后端可执行文件

后端打包输出：

- `dist/backend/backend.exe`

重要约定：

- **后端代码变更后，必须先重建 `backend.exe`，再做 Electron 打包**
- 否则安装包里可能还是旧后端逻辑

### 3.3 Electron 打包顺序

推荐顺序：

```bash
build_backend.bat
cd frontend
npm run build
npm run electron:build
```

### 3.4 打包态运行逻辑

打包后由 Electron 主进程完成：

- 设置 `PYTHON_EXECUTABLE=process.resourcesPath/python/python.exe`
- 启动 `process.resourcesPath/backend/backend.exe`
- 首次启动时由后端对 embedded Python 做最小依赖安装

也就是说：

- **打包产物自带 Python runtime**
- **不要求用户机器预装 Python**
- **首启联网安装的是依赖，不是 Python 本体**

## 4. 最终用户环境

最终用户环境的目标是：**安装后可直接启动，所有用户数据与配置写入用户目录，而不是安装目录。**

### 4.1 用户数据目录

生产环境用户数据默认位于：

- `%APPDATA%/econ-agent/data/`

典型内容：

- `settings.json`
- `memories.db`
- `checkpoints.db`
- `workspace/`
- `tmp/`
- `skills/`

### 4.2 首次启动行为

v0.6.x 在线安装版首次启动时：

1. Electron 拉起后端
2. 后端检查 embedded Python 是否已具备最小启动依赖
3. 若缺失，则使用 `uv` 安装 startup 依赖
4. 若 `uv` 失败，则回退 `pip`
5. 启动完成后进入主界面

### 4.3 首启最小依赖与可选扩展

当前原则：

- 首启只安装**最小必要依赖**
- 统计分析、文档解析、搜索与知识库等能力属于**可选扩展**
- 可选能力缺失时，不应阻塞整个应用启动

对应实现位于：

- `src/api/dependency_installer.py`

## 5. 不会提交 git，但必须统一认知的配置

下面这些内容很多不会提交到仓库，但它们会显著影响开发、打包和用户行为，必须统一认知。

### 5.1 `.env`

来源：仓库根目录 `.env`
模板：`.env.example`

关键变量包括：

- `ANTHROPIC_AUTH_TOKEN`
- `ANTHROPIC_SUB_TOKEN`
- `OPENAI_API_KEY`
- `TAVILY_API_KEY`
- `MINERU_API_KEY`
- `ALIBABA_CLOUD_ACCESS_KEY_ID`
- `ALIBABA_CLOUD_ACCESS_KEY_SECRET`
- `BAILIAN_WORKSPACE_ID`
- `PYTHON_EXECUTABLE`

说明：

- `.env` 通常不提交 git
- 但团队需要对变量名、用途、优先级有统一认知
- 真实值允许因机器/账号不同而不同

### 5.2 用户设置文件

生产环境优先读取：

- `%APPDATA%/econ-agent/data/settings.json`

说明：

- 设置面板保存的值会覆盖 `.env` 中同名配置
- 同一问题排障时，必须确认到底是 `.env` 生效，还是用户目录下 `settings.json` 覆盖了它

### 5.3 开发本地虚拟环境

- `.venv/`

说明：

- 不提交 git
- 但开发时如果 `.venv` 和 `requirements.txt` 不一致，容易出现“本地能跑、打包失败”或“本地误以为用户也有这些依赖”的问题

### 5.4 打包机本地依赖基线

打包机不会把这些环境本体提交到 git，但需要统一基线：

- Python 版本
- Node / npm 版本
- PyInstaller 可用性
- `scripts/setup_python.bat` 是否已执行
- 是否成功生成 `frontend/resources/python/python.exe`
- 是否成功生成 `dist/backend/backend.exe`

### 5.5 本地代理 / 网络条件

首启在线安装依赖时，网络环境会直接影响结果。

特别注意：

- 本地代理可能影响 Python 包安装
- Electron / 前端访问 localhost 时不能错误走系统代理
- 当前项目已在运行时对 localhost 做代理绕过，但排障时仍要确认本机代理状态

## 6. 必须统一的关键环境变量与路径语义

### 6.1 `PYTHON_EXECUTABLE`

语义：当前应用应使用的目标 Python 解释器。

优先级：

1. 显式传入
2. 环境变量 `PYTHON_EXECUTABLE`
3. `sys.executable`

在打包态中，它应指向：

- `resources/python/python.exe`

### 6.2 `ARCSTONE_ECON_USER_DATA`

语义：用户数据目录根路径。

用于保证：

- 安装目录与用户数据目录分离
- 用户数据写到 `%APPDATA%/econ-agent/` 而不是程序安装目录

### 6.3 `ARCSTONE_ECON_INSTALL_ROOT`

语义：安装目录根路径。

用于：

- 读取模板文件
- 识别打包态资源根目录

## 7. 常见失败场景与排障顺序

### 7.1 首启一直卡住

先看：

1. loading 页显示到哪个阶段
2. 是否卡在“检查依赖”还是“安装依赖”还是“健康检查”
3. 用户网络是否可用
4. 阿里云镜像是否可访问

### 7.2 uv 失败

预期行为：

- 自动回退到 `pip`

若仍失败，优先检查：

1. embedded Python 是否可执行
2. `pip` 是否存在
3. 目标解释器是否正确
4. 错误是否来自网络或镜像

### 7.3 开发环境能跑，安装版不能跑

优先检查：

1. 是否只在 `.venv` 里装了依赖，但没有体现在首启安装链路里
2. `dist/backend/backend.exe` 是否重新构建
3. `frontend/resources/python/python.exe` 是否存在
4. Electron 安装包是否实际带上了最新 backend 和 python 资源

### 7.4 依赖已安装但仍提示缺失

优先检查：

1. 安装的依赖是否装到了错误解释器
2. `PYTHON_EXECUTABLE` 是否指向了错误路径
3. 打包态与开发态是否混用了不同解释器

---

## 8. 当前建议的团队执行纪律

1. **开发依赖**、**首启依赖**、**可选扩展依赖**必须分开理解，不能把 `requirements.txt` 直接等同于用户首启安装集合。
2. 用户可见产品逻辑以 **embedded Python + 首启最小依赖安装** 为准，而不是以开发机 `.venv` 为准。
3. 修改后端启动链路时，优先保证：
   - 阶段可见
   - 错误可诊断
   - fallback 明确
4. 若后端代码有变更，打包前必须重建 `backend.exe`。
